# 我为什么做 vault-recall

我做 `vault-recall`，不是因为我想再造一个知识管理工具。

恰恰相反，是因为我越来越觉得：个人知识库真正的问题，已经不是“怎么记录更多东西”了，而是“需要的时候，怎么把以前真实想过的东西找回来”。

Obsidian 用久以后，很容易变成一个很大的仓库。

文件很多，链接很多，目录也越来越完整。看起来很安心。但当你真的要写一篇文章、做一个判断、推进一个项目时，经常会遇到一个很尴尬的问题：

> 我知道我以前想过这个，但我找不到了。

普通关键词搜索在这里不够用。

因为你不一定记得当时用了哪个词。

你现在想搜“发布恐惧”，过去可能写的是“逃避被看见”。

你现在想搜“工具搭太重”，过去可能写的是“工作流反而成了负担”。

你现在想搜“第二大脑召回”，过去可能写的是“外接认知层”。

意思是接近的，但词是不一样的。

这就是大型个人 Vault 的真实痛点：不是没有内容，而是内容沉下去了；不是没有想法，而是旧判断没有被重新召回。

于是你会反复从空白开始。

重新想一遍，重新写一遍，重新组织一遍。时间久了，知识库就从“第二大脑”退化成了“备份仓库”。它保存了很多东西，但很少在关键时刻主动回到你手上。

`vault-recall` 解决的就是这个很具体的问题。

它把 Obsidian 里的 Markdown 扫描出来，切成小块，存进本地 SQLite。默认支持全文搜索；如果配置 embedding，也可以做语义检索。

这样你就可以用模糊的问题去找以前的材料：

```bash
bun vault-recall.ts search "我以前有没有写过技术人用搭环境逃避发布" --semantic
```

它不是新的第二大脑系统。

它只是给你已有的 Vault 加一个很轻的“召回层”。

我刻意没有把它做成复杂工具：

- 不需要后台服务
- 不需要 Obsidian 插件
- 不需要 node_modules
- 不需要安装一堆依赖
- 不会修改你的 Vault
- 数据库放在本地

因为对个人知识库来说，系统越重，越容易变成新的维护负担。

我真正想要的不是再多一个系统，而是在写作、判断、复盘、做项目的时候，能更快把过去的自己叫回来。

记录只是第一步。

召回，才是个人知识库开始真正参与思考的地方。

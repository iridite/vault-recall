# X 长文：vault-recall 发布稿

我最近给自己的 Obsidian 做了一个本地 RAG 小工具。

不是为了折腾知识管理。

是因为我发现，第二大脑最难的不是记录，是找回。

几千篇笔记放在那里，看起来很安心。

但每次写东西前，如果我还是要自己翻目录、猜关键词、回忆文件名，那它本质上还是一个仓库。

仓库有用。

但仓库不会主动帮你接上以前的判断。

这个问题在写自媒体内容时特别明显。

比如我想写：

“技术人为什么总在搭环境，却迟迟不开始发布”

我知道我以前肯定写过类似东西。

但它可能叫：

- 逃避被看见
- 工作流太重
- 发布恐惧
- 低反馈期
- 系统还没准备好
- 体面的拖延

意思接近，关键词不一样。

普通搜索就很容易漏。

所以我做了一个很小的工具：

扫描 Obsidian Markdown。
切块。
存 SQLite。
做 full-text search。
再加 embedding 语义检索。

然后我搜：

“工具搭太重是在逃避发布”

它能把以前散在不同文件里的相关判断捞回来。

这件事让我对 RAG 的理解变了一点。

RAG 对个人知识库最大的价值，不是让 AI 看起来更聪明。

而是帮你重新接上自己。

很多时候你不是没有素材。

你只是忘了自己以前已经想过。

一个判断第一次出现，可能只是日记里一句话。

第二次变成短帖。

第三次变成长文。

第四次可能变成产品判断。

如果没有召回层，它很容易沉下去。

你会一直收集新东西。

但旧判断没有生长。

所以我现在觉得，第二大脑不是资料库。

更准确地说，它应该是一个外接认知层。

记录只是入口。

真正重要的是，当你要写、要判断、要做决策时，它能不能把以前的你叫回来。

我把这个工具整理成了一个开源版，叫 `vault-recall`。

很小。

Bun + SQLite + embedding。
不需要后台服务。
不需要 Obsidian 插件。
不需要 node_modules。
也不用把 Vault 上传到某个云服务。

它只做一件事：

给你的 Obsidian 加一个本地语义召回层。

如果你也经常觉得：

“我以前肯定想过这个，但我找不到了。”

那它可能有用。

个人知识库真正的分水岭，可能不是你记录了多少。

而是你需要输出的时候，它还能不能把你以前真实想过的东西送回来。

不然它只是一个很大的仓库。

看起来很满。

但你每次还是从空白开始。
